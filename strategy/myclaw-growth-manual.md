# MyClaw Growth 战略问题手册 v1.0

_目标：提升留存质量，而不是只追求新增；让不同层级用户都能在可控成本内形成真实使用习惯_
_Last updated: 2026-04-24 | Author: The Doctor 🌀_

---

## 一、先定义真正的问题

MyClaw 当前不是单一的“留存差”问题，而是三个问题叠在一起：

1. **用户价值两极分化**
   一小批头部用户已经找到高价值场景，愿意每天花 $100-$300；大量普通用户没有进入稳定使用状态，很快流失。

2. **产品门槛过高**
   从基础配置、token 获取、skills 安装，到外部工具接入，整条链路都默认用户具备较强的工程调试能力。这对大众用户不成立。

3. **成本结构不匹配**
   OpenClaw 的自由度很高，但这意味着系统默认容易高算力消耗。对于预算敏感用户，如果没有明确的成本边界和智能控制，体验会迅速恶化。

所以，核心矛盾不是“用户不够聪明”，而是：

**MyClaw 还没有把“强大能力”包装成“低门槛、低风险、低认知负担的稳定价值”。**

---

## 二、战略目标：不是让所有人都变成重度用户

正确目标不是把所有用户都推向高消耗，而是让不同层用户进入各自合理的价值密度区间。

### 用户分层与目标区间

| 用户层级 | 典型状态 | 目标日均算力消耗 | 核心目标 |
|----------|----------|------------------|----------|
| 浅层用户 | 偶尔尝试，不稳定使用 | $3-$5 | 活下来，形成轻度依赖 |
| 中度用户 | 已有部分 workflow | $30-$50 | 扩场景，形成周度复用 |
| 重度用户 | 已高度工作流化 | $100-$300 | 提升粘性、扩大钱包份额 |

这三层不是价格带，而是**价值成熟度带**。

因此增长战略要从“统一漏斗”改成“分层经营”：

- 浅层用户要解决的是：能不能快速上手，第一次就看到价值。
- 中度用户要解决的是：能不能把偶发使用变成稳定 workflow。
- 重度用户要解决的是：能不能继续扩使用深度，而不是停留在单一场景。

## 三、问题树：当前留存流失主要死在哪

### 1. 配置死

典型表现：
- 安装完成，但基础配置没跑通
- provider、token、callback、skill 参数太多
- 遇到第一个报错后直接放弃

本质：
- 用户购买的是“能力结果”，却被迫先完成“工程接入”
- 产品把大量 setup 成本前置给了用户

### 2. 技能死

典型表现：
- 装了很多 skills，但都要自己准备 token、API key、环境变量
- 用户以为“安装了 = 能用”，结果实际只是“代码在那儿”
- skills 越多，挫败感越强

本质：
- 当前 skill 生态更像开发者扩展系统，不像普通用户产品
- “可安装”不等于“可激活”，更不等于“可留存”

### 3. 场景死

典型表现：
- 用户知道这东西很强，但不知道该拿来干什么
- 只能想到模糊命令，无法形成日常工作入口
- 首次体验没有击中具体痛点

本质：
- 用户不是需要一个更强的 agent
- 用户需要的是“一个已经被整理好的任务答案”

### 4. 成本死

典型表现：
- 用户刚开始试几次就觉得太贵
- 对 token 消耗没有预期，越用越慌
- 不知道该为哪些任务花大模型成本，哪些不值得

本质：
- 成本不透明 + 路由不智能，会直接打断习惯形成
- 对大多数非头部用户来说，预算焦虑本身就是流失机制

### 5. 习惯死

典型表现：
- 第一次惊艳，第二次没来，第三次就忘了
- 没有明确的周期性触发点
- 用户没有把 MyClaw 植入原有工作流

本质：
- 留存不是满意度函数，而是习惯函数
- 没有固定触发器，再强的能力也会变成偶发玩具

## 四、你已经提出的 3 个方向，判断是对的，但还不够完整

### 方向 1：智能 AI 路由

这是必须做的，而且优先级极高。

因为它同时解决三件事：
- 控成本
- 降低预算焦虑
- 给用户明确的“性价比预期”

但如果只做“模型自动切换”，还不够。你真正要做的是：

**从模型路由升级为任务预算系统。**

即：
- 用户不是在选择模型，而是在选择任务等级
- 系统根据任务等级、时延要求、可靠性要求、历史效果自动路由模型

建议形成三层：
- `Fast`：默认省钱，适合轻问答、整理、转写、分类、摘要
- `Smart`：平衡模式，适合大多数工作任务
- `Max`：顶配模式，适合高价值决策、复杂代码、深度研究

再往前一步，应该加：
- 每日预算保护
- 单任务预算预估
- 超预算自动降级
- 用户可手动锁定某些 workflow 永远走高配

核心思想不是“帮用户省一点钱”，而是：

**把成本从一个不可控黑箱，变成一个可以信任的系统。**

### 方向 2：一键接入生产力工具

这也是必须做的，而且它对留存的价值可能比单纯模型路由还大。

因为一旦接入真实工作软件，MyClaw 就不再只是聊天窗口，而是进入用户真实生产流。

这里的关键不是“支持多少集成”，而是：

**先做最短价值链路的一键接入。**

优先顺序建议：
1. Gmail / Google Calendar / Google Docs
2. Notion
3. Google Drive
4. Slack / Discord
5. Sheets / Airtable

原因：
- 这些工具最接近日常高频生产力
- 一旦接入，最容易形成“收件箱、日程、文档、知识库、协作提醒”这类持续场景

但这里有一个原则必须明确：

**不要先做“开放式接入框架”，先做“官方托管的接入产品”。**

也就是说，普通用户不应该自己去研究 OAuth、token、env、callback URL。
他应该看到的是：
- 点击连接 Google
- 授权
- 立刻获得 3-5 个可直接运行的工作流

不是给他一个 connector；而是给他一个已经配好的生产力能力包。

### 方向 3：数据挖掘与用例化

这个方向是对的，而且这是从“通用 agent”走向“可留存产品”的必要步骤。

因为真正的增长不是从抽象能力出发，而是从高频任务出发。

建议把这件事分成三层：

1. **Use Case 层**
   从活跃用户、头部用户真实行为中提炼任务类型

2. **Solution 层**
   把任务类型整理成可理解的解决方案包

3. **Workflow 层**
   把解决方案拆成可直接运行的固定 workflow、模板、按钮、入口命令

目标不是做一份“案例库”，而是做一套：

**用户看到一个场景，就可以立刻运行，而不是先理解半小时。**

## 五、你还没明确说出来，但必须补上的 4 个模块

### 模块 1：首日激活设计

大部分流失，不发生在第 30 天，而发生在第 30 分钟。

所以必须单独设计 `Day 0` 体验。

首日激活的目标不是让用户理解产品全貌，而是让他在 5 分钟内完成一次成功闭环：
- 一个真实任务被完成
- 用户知道下一步还能做什么
- 用户感觉成本可控

首日激活页不应该是“欢迎使用 MyClaw”，而应该是：
- 你想先连接什么：Google / Notion / 无需连接先试用
- 你想先解决什么：邮件整理 / 日程管理 / 文档总结 / 信息搜集 / 内容生成
- 你的预算偏好：省钱优先 / 平衡 / 顶配

也就是说，首屏不该是开放输入框，而该是：

**场景入口 + 成本选择 + 一键开始。**

### 模块 2：默认工作流产品化

如果一切都要用户自己搭，绝大多数人永远不会完成搭建。

所以你需要一批官方默认 workflow，作为产品骨架。

建议优先做 10 个：
- 今日邮件清理
- 今日日程总结
- 会议纪要整理
- 网页/文档快速总结
- Notion 知识库问答
- 每日行业情报摘要
- 内容改写与发布草稿
- 销售/客服消息草拟
- 待办事项拆解
- 周报生成

这些 workflow 的要求不是炫技，而是：
- 3 分钟内跑通
- 明确输入输出
- 成本可预测
- 可重复使用

真正能留存用户的不是 Agent 的开放能力，而是这些可重复调用的默认骨架。

### 模块 3：主动触发机制

留存的本质不是“用户会不会回来”，而是“系统有没有理由再次出现”。

所以 MyClaw 不能只做被动响应，还要做适度主动。

主动不是乱推送，而是围绕高价值触发点：
- 早晨邮件摘要
- 会前 briefing
- 会后纪要整理
- 每日晚间工作总结
- 行业监控更新
- 某个 workflow 运行异常提醒

一旦用户允许这些触发，MyClaw 才会进入生活节奏，而不是停留在工具栏里。

这里的关键是：

**从“聊天产品”变成“周期性工作代理”。**

### 模块 4：分层运营与产品面分离

现在最危险的错误，是把所有用户都塞进同一个界面、同一套能力体系。

实际上应该分成三种产品面：

1. **新手面**
   默认只展示少量入口、少量 workflow、少量接入选项，核心是低认知负担

2. **进阶面**
   允许用户保存 workflow、管理预算、连接更多工具、查看历史自动化结果

3. **专家面**
   保留 OpenClaw 原生自由度，包括复杂 skill、代码、深度 agent 配置

这不是功能阉割，而是认知隔离。

大多数用户不需要完整自由，他们需要的是：

**被组织过的自由。**

## 六、真正正确的产品顺序

如果顺序错了，团队会陷入“做了很多能力，但留存还是没起来”。

正确顺序应该是：

1. **先控成本，再扩使用**
   没有预算边界，普通用户不敢用；不敢用，就不会留存。

2. **先做官方托管接入，再做开放生态接入**
   先把 Google/Notion 这种高频场景打磨成傻瓜式，再谈开放给更多开发者和 skills。

3. **先做默认 workflow，再做抽象 agent 能力教育**
   用户先要成功，才会理解平台。

4. **先做头部用例拆解，再做广义场景库**
   真正的高留存场景来自真实数据，不来自头脑风暴。

5. **先做触发和复用，再追求首日惊艳**
   留存取决于第二次、第三次能不能自然发生。

## 七、建议的战略主轴

如果要把这一阶段的 MyClaw Growth 压缩成一句话，我会这样定义：

**把 OpenClaw 从“一个很强但很难驾驭的能力容器”，重构成“不同预算用户都能稳定获得结果的工作流系统”。**

这句话对应四个主轴：

### 主轴 A：成本可控
- 智能路由
- 预算上限
- 任务分级
- 成本预估

### 主轴 B：接入够傻瓜
- 官方托管连接器
- 减少 token / env / callback 暴露
- 接入后立刻附赠可运行 workflow

### 主轴 C：场景够具体
- 头部用户数据挖掘
- 行业 use case 包装
- workflow 模板化
- solution library

### 主轴 D：复用够自然
- 主动提醒
- 定时任务
- 周期工作流
- 历史结果可复跑

## 八、优先级判断：什么先做，什么后做

### P0：必须立刻推进
- 智能路由 + 预算系统
- Google / Notion 一键接入
- 首批 10 个官方默认 workflow
- 新手面产品化入口

### P1：紧接着做
- 头部用户 use case 数据挖掘
- 行业 solution library
- 主动触发机制
- workflow 保存、复跑、参数化

### P2：后续放大器
- 更广泛的 app / skill 接入生态
- 更多第三方 connector
- 行业化模板市场
- 生态创作者分发体系

这个顺序背后的逻辑是：

**先解决留存，再解决扩张；先解决可用，再解决丰富。**

## 九、增长团队需要盯住的核心指标

不要只看 DAU、收入和实例数。要看能否穿透到真正的留存机制。

建议重点盯这几类：

### 激活指标
- 新用户 24 小时内是否完成首次成功任务
- 新用户 24 小时内是否连接至少一个生产力工具
- 新用户 24 小时内是否运行至少一个默认 workflow

### 留存指标
- D1 / D7 / D30 留存
- 有接入工具用户 vs 无接入工具用户的留存差
- 有 workflow 保存行为用户 vs 无保存行为用户的留存差

### 成本指标
- 不同层用户的日均算力消耗是否落入目标区间
- 首周用户中超预算用户占比
- 智能路由带来的平均任务成本下降幅度

### 产品化指标
- 默认 workflow 的使用率与复用率
- 每个连接器授权完成率
- 每个 use case 页面到实际运行的转化率

### 结构性指标
- 头部用户收入占比是否持续过高
- 中度用户占比是否提升
- 浅层用户中成功跨入中度用户的转化率

## 十、最后的判断

你现在看到的问题，本质上不是“怎么让更多人学会 OpenClaw”，而是：

**怎么把 OpenClaw 的能力，重新包装成适合大多数人持续使用的产品形态。**

头部用户已经证明了价值存在。
现在真正的战略任务，不是继续证明 MyClaw 很强，而是证明：

1. 普通用户也能上手
2. 普通预算也能长期用
3. 普通工作流也能不断复用

只要这三件事成立，留存会自然改善，CAC 才有继续放大的意义。

否则，新增越大，只会把流失漏斗放得更大。
