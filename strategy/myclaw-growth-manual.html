这份文档按内部执行版来写。目标不是展示观点，而是把留存问题、执行顺序和优先级讲清楚。

真正的问题不是用户不够聪明，而是 MyClaw 还没有把强能力包装成低门槛、低风险、低认知负担的稳定价值。

后面所有判断，都围绕这句话展开。

Segmentation

分层目标：不是让所有人都变重度

正确目标不是把所有用户都推向高消耗，而是让不同层级用户进入各自合理的价值密度区间。经营逻辑不是统一漏斗，而是分层经营。

用户层级 典型状态 目标日均算力 核心任务
浅层用户 偶尔尝试，不稳定使用 $3-$5 先活下来，形成轻度依赖
中度用户 已有部分 workflow，但复用不稳定 $30-$50 从偶发使用走向周度复用
重度用户 已高度工作流化 $100-$300 扩场景、提粘性、扩大钱包份额

执行含义：浅层用户要第一次成功，中度用户要 workflow 固化，重度用户要继续扩深度。三层的产品面和运营策略不能混在一起。

Playbook

用户分层打法：三层用户要三套经营逻辑

分层不是为了写一张表，而是为了决定不同用户该被推向什么产品动作、什么运营动作，以及什么升级路径。

浅层用户：目标是活下来，不是立刻榨出高 ARPU

这一层用户的核心问题不是需求不足，而是门槛太高、成功太慢、预算太不确定。对这层人，最重要的是第一次成功和第二次返回。

浅层用户打法
  • 默认走省钱路由，不让用户第一周产生预算惊吓
  • 默认只暴露少量入口，不暴露复杂自由度
  • 优先推无需配置或轻配置的官方 workflow
  • 运营目标是让用户完成第一次成功任务和第一次复用

中度用户：目标是从偶发使用，走向固定 workflow

这一层用户已经相信价值存在，但还没有形成稳定工作流。他们流失通常不是因为不懂，而是因为没有完成 workflow 固化。

中度用户打法
  • 推动连接 Google / Notion 等真实生产力工具
  • 推动保存、复跑、参数化已有 workflow
  • 推动从单场景扩到相邻场景，比如邮件 → 日程 → 周报
  • 运营目标是让用户形成每周至少一次固定复用

重度用户：目标是扩深度，不是只维持现状

这一层用户已经给出高价值反馈。他们的问题不是能不能用，而是能不能在更多场景里继续花更多时间、更多预算，并且不流向别处。

重度用户打法
  • 允许锁定高配模型和更高预算上限
  • 优先给他们开放更深的 workflow 编排和自动化能力
  • 优先挖他们的真实行为数据，反向提炼官方模板
  • 运营目标是扩场景、提高钱包份额、降低替代风险

关键点：浅层用户关注第一次成功，中度用户关注 workflow 固化，重度用户关注场景扩张。三层共用一套界面和一套推进逻辑，会把效率全部吃掉。

Activation

7天激活链路：留存不是首日惊艳，而是 7 天内形成节奏

新用户留存不是 onboarding 问题，而是节奏问题。真正要拆的是前 7 天里，系统如何把一次偶然尝试变成连续使用。

Day 0：第一次成功

用户刚进来时，目标不是理解平台，而是完成一个真实任务。这个阶段最怕的不是功能少，而是配置多。

Day 0 目标
  • 用户完成一次真实任务闭环
  • 用户感受到成本可控
  • 用户知道下一步可以做什么

Day 1-3：让第二次使用自然发生

很多产品的问题不是没有首次成功，而是第二次使用没有被设计出来。用户离开后没有被系统重新拉回，就会迅速忘掉。

Day 1-3 目标
  • 推动连接第一个真实生产力工具
  • 推动运行第二个相关 workflow
  • 通过提醒或建议，把使用从一次性动作变成连续动作

Day 4-7：形成第一个固定 workflow

7 天内如果还没有出现可复跑 workflow，留存大概率会掉。真正的分水岭不是用户是否满意，而是有没有把一个动作固化下来。

Day 4-7 目标
  • 保存第一个 workflow
  • 完成至少一次复跑
  • 建立一个定期触发点，比如每天或每周一次

执行判断：如果一个新用户在 7 天内没有连接至少一个工具、没有完成至少一次复跑、没有形成一个固定触发点，那么这个用户大概率还没有进入真正的留存轨道。

Problem Tree

流失机制：当前主要死在五个地方

1. 配置死

用户想买结果，却先被迫做工程接入。setup 成本被前置，第一次体验直接被拖垮。

2. 技能死

当前 skill 生态更像开发者扩展系统，不像普通用户产品。可安装，不等于可激活，更不等于可留存。

3. 场景死

用户知道它很强，但不知道拿来干什么。没有被整理好的任务入口，就不会形成日常使用。

4. 成本死

成本不透明、路由不智能时，预算焦虑本身就是流失机制。用户不是不用，而是不敢用。

5. 习惯死

第一次惊艳不会自动变成留存。没有周期性触发点，MyClaw 就进不了用户的工作节奏。

Three Directions

三条主线：方向是对的，但要更产品化

Direction 1

智能 AI 路由，要升级成任务预算系统

如果只做模型自动切换，还不够。用户不该选模型，而该选任务等级。系统再根据任务等级、时延要求和历史效果自动路由。

  • Fast：轻问答、整理、分类、摘要，默认省钱
  • Smart：覆盖大多数工作任务，平衡质量与成本
  • Max：高价值决策、复杂代码、深度研究

要补的不是小优化，而是预算保护、单任务预估、超预算降级、特定 workflow 高配锁定。目标不是省一点钱，而是把成本从黑箱变成系统承诺。

Direction 2

一键接入生产力工具，不要先做开放 connector

普通用户不应该先研究 OAuth、token、env 和 callback URL。他应该直接看到：点击连接 Google，授权，立刻拿到 3 到 5 个能运行的 workflow。

优先顺序建议还是一样：

  1. Gmail / Google Calendar / Google Docs
  2. Notion
  3. Google Drive
  4. Slack / Discord
  5. Sheets / Airtable

关键不是支持多少集成，而是先做最短价值链路的一键接入，再谈开放生态。

Direction 3

数据挖掘与用例化，终点不是案例库，而是 workflow

真正的增长不是从抽象能力出发，而是从高频任务出发。头部用户的数据价值，不在于证明 MyClaw 很强，而在于告诉你哪些任务值得产品化。

  • Use Case 层：提炼真实任务类型
  • Solution 层：包装成普通用户看得懂的解决方案
  • Workflow 层：落成按钮、模板、入口命令和可复跑流程
Missing Modules

缺失模块：真正决定留存的四件事

1. 首日激活设计

大部分流失不发生在第 30 天，而发生在第 30 分钟。首日目标只有一个：让用户在 5 分钟内完成一次真实任务闭环。

2. 默认 workflow 产品化

如果一切都要用户自己搭，绝大多数人永远不会完成搭建。官方必须先给出一批可反复调用的骨架。

3. 主动触发机制

MyClaw 不能只在被问到时出现。它需要围绕高价值时刻主动出现，进入用户的工作节奏。

4. 分层产品面

所有用户都塞进同一套能力体系，是最危险的错误。要拆成新手面、进阶面、专家面，做认知隔离。

Sequence

执行顺序：顺序错了，团队会一直忙但结果很差

这里最重要的不是做什么，而是先后顺序。先后顺序决定留存能不能起来。

  1. 先控成本，再扩使用
  2. 先做官方托管接入，再做开放生态接入
  3. 先做默认 workflow，再做抽象 agent 能力教育
  4. 先做头部用例拆解，再做广义场景库
  5. 先做触发和复用，再追求首日惊艳

一句话概括：先解决留存，再解决扩张；先解决可用，再解决丰富；先做成产品，再做成生态。

Priorities

优先级：P0 / P1 / P2 应该怎么排

P0

必须立刻推进

  • 智能路由 + 预算系统
  • Google / Notion 一键接入
  • 首批 10 个官方默认 workflow
  • 新手面产品化入口
P1

紧接着做

  • 头部用户用例挖掘
  • 行业 solution library
  • 主动触发机制
  • workflow 保存、复跑、参数化
P2

后续放大器

  • 更广泛的 app / skill 生态
  • 更多第三方 connector
  • 行业模板市场
  • 创作者分发体系
Metrics

关键指标：增长团队真正该盯什么

不要只看 DAU、收入和实例数。要看是否真正穿透到激活、复用、预算控制和结构变化。

激活指标

  • 24 小时内是否完成首次成功任务
  • 24 小时内是否连接至少一个生产力工具
  • 24 小时内是否运行至少一个默认 workflow

留存指标

  • D1 / D7 / D30 留存
  • 有工具接入用户 vs 无接入用户的留存差
  • 有 workflow 保存行为用户 vs 无保存行为用户的留存差

成本指标

  • 不同层用户是否落入目标预算区间
  • 首周超预算用户占比
  • 智能路由带来的平均任务成本下降

结构性指标

  • 头部用户收入占比是否过高
  • 中度用户占比是否提升
  • 浅层用户跨入中度用户的转化率
Conclusion

最后的判断

MyClaw Growth 下一阶段的核心，不是再卖“最强 Agent”，而是卖“在你的预算内，持续替你完成工作的工作流系统”。

头部用户已经证明了价值存在。下一阶段的任务，不是继续证明 MyClaw 很强，而是把这个价值翻译成普通用户也能长期复用的产品结构。
