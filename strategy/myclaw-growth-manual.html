Segmentation

不是让所有人都变成重度用户

正确目标不是把所有用户都推向高消耗，而是让不同层级用户进入各自合理的价值密度区间。你要经营的不是统一漏斗，而是三种不同成熟度的人群。

用户层级 典型状态 目标日均算力 核心任务
浅层用户 偶尔尝试，不稳定使用 $3-$5 先活下来，形成轻度依赖
中度用户 已有部分 workflow，但复用不稳定 $30-$50 从偶发使用走向周度复用
重度用户 已高度工作流化 $100-$300 扩场景、提粘性、扩大钱包份额

关键点：这三层不是价格带，而是价值成熟度带。浅层用户要的是第一次成功，中度用户要的是工作流固化，重度用户要的是继续扩深度。

Problem Tree

当前留存主要死在五个地方

1. 配置死

用户购买的是能力结果，却被迫先完成工程接入。provider、token、callback、env 这些复杂度都被前置了，于是第一次体验被 setup 成本吞掉。

2. 技能死

装了很多 skills，不等于真的能用。当前生态更像开发者扩展系统，不像普通用户产品。可安装，不等于可激活，更不等于可留存。

3. 场景死

用户知道这东西很强，但不知道拿来干什么。面对开放输入框和抽象能力说明，大多数人都无法自然形成日常任务入口。

4. 成本死

预算焦虑本身就是流失机制。成本不透明、路由不智能时，用户会迅速产生“越用越亏”的心理，这会直接打断习惯形成。

5. 习惯死

第一次惊艳不会自动变成长期留存。没有周期性触发点，MyClaw 就只会停留在工具栏里，而不是进入用户每天的节奏中。

Three Directions

你提出的三条主线，判断是对的

Direction 1

智能 AI 路由，要升级成任务预算系统

如果只做模型自动切换，还不够。真正应该发生的变化是：让用户不是在选模型，而是在选任务等级。系统再根据任务等级、时延要求、可靠性要求和历史效果自动路由。

  • Fast：轻问答、整理、分类、摘要，默认省钱
  • Smart：覆盖大多数工作任务，平衡质量与成本
  • Max：高价值决策、复杂代码、深度研究

真正要补的是预算保护、单任务预估、超预算降级、特定 workflow 高配锁定。这不是“省一点钱”，而是把成本从黑箱变成可被信任的系统。

Direction 2

一键接入生产力工具，不要先做开放 connector

普通用户不应该先研究 OAuth、token、env 和 callback URL。他应该看到的是：点击连接 Google，授权，立刻拿到 3 到 5 个能直接运行的工作流。

优先顺序建议还是一样：

  1. Gmail / Google Calendar / Google Docs
  2. Notion
  3. Google Drive
  4. Slack / Discord
  5. Sheets / Airtable

关键不是支持多少集成，而是先做最短价值链路的一键接入。

Direction 3

数据挖掘与用例化，终点不是案例库，而是 workflow

真正的增长不是从抽象能力出发，而是从高频任务出发。头部用户的数据价值，不在于证明 MyClaw 很强，而在于告诉你哪些任务值得产品化。

  • Use Case 层：提炼真实任务类型
  • Solution 层：包装成普通用户看得懂的解决方案
  • Workflow 层：落成按钮、模板、入口命令和可复跑流程
Missing Modules

真正缺的，是四个决定留存的模块

1. 首日激活设计

大部分流失不发生在第 30 天，而发生在第 30 分钟。首日目标不是让用户理解产品全貌，而是让他在 5 分钟内完成一次真实任务闭环。

2. 默认 workflow 产品化

如果一切都要用户自己搭，绝大多数人永远不会完成搭建。官方必须先给出一批可反复调用的骨架，例如今日邮件清理、今日日程总结、会议纪要整理、周报生成。

3. 主动触发机制

MyClaw 不能只在被问到时出现。它需要围绕高价值时刻主动出现，比如早晨邮件摘要、会前 briefing、会后纪要整理、异常提醒。

4. 分层产品面

所有用户都塞进同一套能力体系，是最危险的错误。应该拆成新手面、进阶面、专家面，做认知隔离，而不是一股脑开放全部自由度。

Sequence

真正正确的产品顺序

如果顺序错了，团队会一直很忙，但留存起不来。这里最重要的不是做什么，而是先后顺序。

  1. 先控成本，再扩使用
  2. 先做官方托管接入，再做开放生态接入
  3. 先做默认 workflow，再做抽象 agent 能力教育
  4. 先做头部用例拆解，再做广义场景库
  5. 先做触发和复用，再追求首日惊艳

一句话概括：先解决留存，再解决扩张；先解决可用，再解决丰富。

Priorities

P0 / P1 / P2 应该怎么排

P0

必须立刻推进

  • 智能路由 + 预算系统
  • Google / Notion 一键接入
  • 首批 10 个官方默认 workflow
  • 新手面产品化入口
P1

紧接着做

  • 头部用户用例挖掘
  • 行业 solution library
  • 主动触发机制
  • workflow 保存、复跑、参数化
P2

后续放大器

  • 更广泛的 app / skill 生态
  • 更多第三方 connector
  • 行业模板市场
  • 创作者分发体系
Metrics

增长团队真正该盯的指标

不要只看 DAU、收入和实例数。要看是否真正穿透到激活、复用、预算控制和结构变化。

激活指标

  • 24 小时内是否完成首次成功任务
  • 24 小时内是否连接至少一个生产力工具
  • 24 小时内是否运行至少一个默认 workflow

留存指标

  • D1 / D7 / D30 留存
  • 有工具接入用户 vs 无接入用户的留存差
  • 有 workflow 保存行为用户 vs 无保存行为用户的留存差

成本指标

  • 不同层用户是否落入目标预算区间
  • 首周超预算用户占比
  • 智能路由带来的平均任务成本下降

结构性指标

  • 头部用户收入占比是否过高
  • 中度用户占比是否提升
  • 浅层用户跨入中度用户的转化率
Conclusion

最后的判断

MyClaw Growth 下一阶段的核心，不是再卖“最强 Agent”，而是卖“在你的预算内，持续替你完成工作的工作流系统”。

头部用户已经证明了价值存在。现在真正的战略任务，不是继续证明 MyClaw 很强，而是证明普通用户也能上手、普通预算也能长期用、普通工作流也能不断复用。
